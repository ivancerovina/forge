# Forge — AI Agent Reference

Forge is a CLI tool for managing local development projects with Docker Compose, Traefik reverse proxy, and optional Cloudflare tunnel integration.

## Architecture Overview

Forge manages a **system layer** (shared Traefik proxy + Docker network) and per-project **environments** (Docker Compose services with domain routing).

```
User projects                        System infrastructure
┌──────────────┐                    ┌──────────────────────┐
│ .forgerc.json │─ forge project start ▶│ docker compose up    │
│ compose.yaml  │                    │ forge-network connect│
└──────────────┘                    └──────────────────────┘
       │                                      │
       │ forge project bind                   ▼
       ▼                            ┌──────────────────────┐
┌──────────────┐                    │ Traefik (ports 80/443)│
│ /etc/hosts   │                    │ Dynamic routing       │
│ traefik/*.yml│                    │ Optional: cloudflared │
└──────────────┘                    └──────────────────────┘
```

## Directory Layout

### Global (`~/.forge/`)

| File | Purpose |
|------|---------|
| `config.json` | Global config: `cloudflare_domain`, `cloudflare_tunnel` |
| `projects.json` | Array of absolute paths to registered projects |
| `docker-compose.yml` | System Traefik stack (+ cloudflared if tunnel enabled) |
| `cf-config.yml` | Cloudflared ingress config |
| `certs/local.pem`, `certs/local-key.pem` | TLS certs (via mkcert) |
| `traefik/_tls.yml` | Traefik TLS config |
| `traefik/<code>.yml` | Per-project Traefik routing (generated by `forge project bind`) |
| `schemas/forgerc.schema.json` | JSON Schema for `.forgerc.json` |

### Per-project (`.forgerc.json`)

Located in the project root. Auto-discovered by walking up from cwd, stopping at `.git` boundary or home directory.

## `.forgerc.json` Full Schema

```json
{
  "$schema": "file:///home/<user>/.forge/schemas/forgerc.schema.json",
  "name": "My Project",
  "description": "Optional description",
  "code": "my-project",
  "environment": {
    "compose_file": "docker-compose.yml",
    "hooks": {
      "pre_start": ["echo starting"],
      "post_start": [],
      "pre_stop": [],
      "post_stop": [],
      "pre_destroy": [],
      "post_destroy": []
    },
    "alias": [
      {
        "container": "myproject-frontend",
        "port": 5173,
        "alias": null,
        "https": true,
        "cloudflare": true
      },
      {
        "container": "myproject-backend",
        "port": 3000,
        "alias": "backend",
        "path": "/api",
        "forward_pathname": false,
        "target_path": "/v2",
        "https": true,
        "cloudflare": false
      }
    ]
  }
}
```

### Field Reference

| Field | Required | Type | Description |
|-------|----------|------|-------------|
| `name` | yes | string | Human-readable project name (non-empty, single line) |
| `code` | yes | string | URL-safe identifier: `^[a-zA-Z0-9]([a-zA-Z0-9-]*[a-zA-Z0-9])?$` |
| `description` | no | string | Project description |
| `environment.compose_file` | no | string | Relative path to compose file. Empty = auto-detect (`compose.yaml` > `compose.yml` > `docker-compose.yml` > `docker-compose.yaml`) |
| `environment.hooks.*` | no | string[] | Shell commands run via `sh -c` before/after compose operations |
| `environment.alias` | no | array | Array of alias entries (legacy map format auto-migrated) |

### Alias Entry Fields

| Field | Required | Type | Default | Description |
|-------|----------|------|---------|-------------|
| `container` | yes | string | — | Docker container name (`container_name` or service name from compose file). Deprecated `service` key still accepted. Pattern: `^[a-zA-Z0-9][a-zA-Z0-9_-]*$` |
| `port` | yes | int (1-65535) | — | Container port to route to |
| `alias` | no | string or null | null | `null` = `<code>.test`, `"sub"` = `sub.<code>.test` |
| `path` | no | string | — | Frontend path prefix for routing (e.g., `/api`). Must start with `/`, no trailing `/`. Pattern: `^/[a-zA-Z0-9/_-]+$` |
| `forward_pathname` | no | bool | false | Forward path prefix to backend as-is (default: strip prefix before forwarding) |
| `target_path` | no | string | — | Backend target path appended to the service URL (e.g., `/test`). Same pattern as `path`. |
| `https` | no | bool | true | Enable HTTPS (requires mkcert certs) |
| `cloudflare` | no | bool | false | Also create a public Traefik router via Cloudflare tunnel |

### Domain Routing Logic

Given `code = "myapp"`:

| `alias` | `path` | Local domain |
|---------|--------|-------------|
| `null` | — | `myapp.test` |
| `"api"` | — | `api.myapp.test` |
| `null` | `"/api"` | `myapp.test/api` (with StripPrefix unless `forward_pathname: true`) |
| `"api"` | `"/v2"` | `api.myapp.test/v2` (with StripPrefix unless `forward_pathname: true`) |

With Cloudflare tunnel domain `dev.example.com` and `cloudflare: true`:

| `alias` | Public domain |
|---------|--------------|
| `null` | `myapp.dev.example.com` |
| `"api"` | `api.myapp.dev.example.com` |

### Backend Target Path (`target_path`)

- `path` = **frontend** — Traefik matches `PathPrefix` on incoming requests
- `target_path` = **backend** — appended to the Traefik service URL (`http://container:port/target_path`)

These compose: a request to `myapp.test/api/users` with `path: "/api"`, `forward_pathname: false`, `target_path: "/v2"` strips `/api` then forwards to `http://container:port/v2/users`.

| `path` | `forward_pathname` | `target_path` | Request to `/api/users` → backend receives |
|--------|--------------------|---------------|---------------------------------------------|
| `/api` | false | — | `/users` |
| `/api` | true | — | `/api/users` |
| `/api` | false | `/v2` | `/v2/users` |
| — | — | `/test` | `/test/` (all requests forwarded to `/test` base) |

## All Commands

### System

| Command | Description |
|---------|-------------|
| `forge setup` | Create Docker network + start Traefik proxy. Idempotent. (`forge init` is a hidden alias) |

### Project Lifecycle

| Command | Description |
|---------|-------------|
| `forge project init` | Create `.forgerc.json` (interactive or with `-t`/`-c` flags) |
| `forge project init -t "Name" -c "code"` | Non-interactive init. Optional: `-d`, `--register`, `--force` |
| `forge project start` | `docker compose up -d` + connect to forge-network + show status |
| `forge project stop [all\|<name>]` | Stop project. No arg = cwd, `all` = all registered, `<name>` = by project name |
| `forge project destroy` | `docker compose down` |
| `forge project info` | Show project details, service states, and alias overview |

`forge start`, `forge stop`, `forge destroy` still work as hidden top-level aliases.

### Project Management

| Command | Description |
|---------|-------------|
| `forge project register [path]` | Add project to `~/.forge/projects.json` |
| `forge project unregister [path]` | Remove project from registry |
| `forge project list` | List all registered projects with names and paths |

### Domain Binding

**NEVER run `forge project bind` or `forge project unbind` from an agent context.** These commands require elevated permissions (`sudo` for `/etc/hosts`) and will prompt for a password interactively, which agents cannot provide. Alias changes (`alias add`/`alias remove`) automatically trigger bind/unbind, so manual binding is rarely needed.

| Command | Description |
|---------|-------------|
| `forge project bind` | Generate Traefik routing + add `/etc/hosts` entries (requires sudo — user only) |
| `forge project unbind` | Remove Traefik config + `/etc/hosts` entries (requires sudo — user only) |

### Alias Management

| Command | Description |
|---------|-------------|
| `forge project alias add <service> --port <port>` | Add service alias. Auto-binds domains. Flags: `--alias`, `--path`, `--forward-pathname`, `--target-path`, `--http`, `--cloudflare`, `--force` |
| `forge project alias remove <service>` | Remove a service alias. Auto-binds/unbinds domains. |
| `forge project alias info [service]` | Show alias details (one or all) |

All alias commands support interactive mode (run without arguments). `alias add` and `alias remove` automatically run bind/unbind after changes.

### Cloudflare Tunnel

| Command | Description |
|---------|-------------|
| `forge tunnel init` | Enable tunnel (requires `$CLOUDFLARE_TUNNEL_TOKEN`) |
| `forge tunnel stop` | Disable tunnel, remove cloudflared container |
| `forge tunnel set-domain <domain>` | Set base domain (e.g., `dev.example.com`) |
| `forge tunnel info` | Show tunnel config and container state |

## Common AI Tasks

### Adding a container alias to `.forgerc.json`

Edit the `environment.alias` array. Each entry needs a `container` field matching `^[a-zA-Z0-9][a-zA-Z0-9_-]*$`. The deprecated `service` key is still accepted but triggers a warning.

```json
"alias": [
  { "container": "myapp-web", "port": 3000, "alias": null },
  { "container": "myapp-api", "port": 8080, "alias": "api", "path": "/v1", "target_path": "/test" }
]
```

The legacy map format (keys as service names) and deprecated `service` key are still read but automatically migrated on write.

If editing `.forgerc.json` directly (not via `forge project alias add`), the user must run `forge project bind` manually to apply routing. Agents should not run bind commands.

### Adding lifecycle hooks

```json
"hooks": {
  "pre_start": ["npm install", "npm run build"],
  "post_start": ["echo 'Services are up'"],
  "pre_stop": [],
  "post_stop": []
}
```

Hooks are shell commands run sequentially via `sh -c` in the project directory.

### Setting a custom compose file path

```json
"environment": {
  "compose_file": "docker/docker-compose.dev.yml"
}
```

Path is relative to the directory containing `.forgerc.json`. If omitted, forge auto-detects from candidates: `compose.yaml`, `compose.yml`, `docker-compose.yml`, `docker-compose.yaml`.

### Enabling Cloudflare for a service

1. Ensure tunnel is initialized: `forge tunnel init`
2. Set domain: `forge tunnel set-domain dev.example.com`
3. Add `"cloudflare": true` to the alias entry (or use `forge project alias add <svc> --port <port> --cloudflare`)
4. User runs `forge project bind` to apply (auto-bind handles this when using `alias add`)

## Validation Rules Summary

| Field | Rule |
|-------|------|
| Project name | Non-empty, single line |
| Project code | `^[a-zA-Z0-9]([a-zA-Z0-9-]*[a-zA-Z0-9])?$` |
| Service name | `^[a-zA-Z0-9][a-zA-Z0-9_-]*$` |
| Port | Integer 1-65535 |
| Path | Starts with `/`, no trailing `/`, matches `^/[a-zA-Z0-9/_-]+$` |
| Alias subdomain | Empty (= index) or matches code pattern |
| Domain | Non-empty, 2+ dot-separated labels, each matches code pattern |

## Important Behaviors

- **Walk-up discovery:** Most commands find `.forgerc.json` by walking up from cwd, stopping at `.git` or home directory. No need to be in the project root.
- **No root:** Forge refuses to run as root. Commands needing elevation (writing `/etc/hosts`) use `sudo` internally for just that operation.
- **`$schema` auto-set:** Every `WriteForgeRC` call adds the `$schema` field automatically. Don't remove it.
- **Registry is just paths:** `projects.json` is an array of absolute directory paths. Project metadata lives only in each `.forgerc.json`.
- **Hosts markers:** `/etc/hosts` entries are tagged with `# forge:<project-code>` for clean add/remove.
- **Legacy format:** `environment.commands` (with `start`/`stop`/`destroy` arrays) is deprecated. Forge still runs them but prints a warning. New configs should use `hooks` + native compose.
